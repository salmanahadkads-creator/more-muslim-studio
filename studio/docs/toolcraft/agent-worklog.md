### Iteration 11 — Caption/audio sync: lock the render clock to the audio clock

### Iteration 16 — Exported MP4 played silent: a doubly-nested AAC decoder config

- Request: "audiogram export doesn't have audio", with the silent `more-muslim-audiogram.mp4` plus the source `.mp3`/`.srt` (the Japan halal-boom stringout) attached.
- Task type: export/muxing bug fix in the MP4 audio path.
- User-visible result: exported MP4s now carry audible sound. Previously every MP4 export was silent — and misleadingly so, because the file *looked* complete: a `soun` track, AAC codec, 3,106 samples, 716 KB of audio data and a 66.26s duration that matched the video exactly.
- Source/reference checked: parsed the silent MP4's box tree directly rather than trusting the symptom. The audio track, sample count, duration and `mp4a` sample entry (2ch / 16-bit / 48 kHz) were all correct, which ruled out "no audio was encoded". The fault was one level deeper, in the `esds` decoder config: its DecoderSpecificInfo (tag 0x05) was 39 bytes beginning `03 80 80 80 22` — an entire nested ES_Descriptor where the bare AudioSpecificConfig belongs. Then read `mp4-muxer@5.2.2` `build/mp4-muxer.js` `esds()`, which builds the whole descriptor tree itself and splices `decoderConfig.description` in at the tag-5 slot, confirming the muxer expects a BARE config while Chrome's AAC encoder had handed back a full ES_Descriptor. Buried inside the wrapped blob was the correct config, `1190` (AAC-LC, 48 kHz, stereo).
- Reference inputs: the user's actual silent export and its real source media, used both to diagnose and to verify — the malformed 39-byte config became the unit-test fixture verbatim.
- Docs/contracts read: none newly required (export-path bug, no schema/contract surface change).
- Contract rules applied: the normalisation happens at the app↔muxer boundary in `export-audiogram.ts` (no runtime/`src/toolcraft` edit, no new sanctioned delta); it is scoped to the MP4/AAC path so the WebM/Opus description is untouched.
- Decision: unwrap `decoderConfig.description` to the bare AudioSpecificConfig before handing metadata to the muxer, via a general MPEG-4 descriptor walker that recurses ES_Descriptor (3) → DecoderConfigDescriptor (4) → DecoderSpecificInfo (5), honouring the variable-length size encoding and the optional ES_Descriptor fields (streamDependence / URL / OCR flags). A description that is already bare is returned untouched, so this is safe across browsers that already emit the correct form — the guard is `bytes[0] !== 0x03`, which also naturally passes Opus ("OpusHead") through.
- Alternatives rejected: hard-coding the two known-good bytes `1190` (would break the moment sample rate or channel count changes — the config encodes both); patching or pinning mp4-muxer (the muxer's expectation is reasonable and documented by its own code; the mismatch is Chrome's wrapped output); re-muxing with ffmpeg post-export (heavyweight and does not fix the actual defect).
- State/output mapping: `AudioEncoder` output metadata → `toBareAudioSpecificConfig(description)` → `addAudioChunk(chunk, normalizedMetadata)` → mp4-muxer writes `esds` with a valid 2-byte ASC → players initialise the AAC decoder → audible export.
- Files changed: src/app/export-audiogram.ts (descriptor walker + `toBareAudioSpecificConfig` + normalisation in the encoder output callback), src/app/export-audio-config.test.ts (new).
- Verification: pnpm typecheck passed; pnpm test passed fully (324/324); pnpm build passed.
